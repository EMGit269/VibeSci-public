# Research Code Manager

## 任务目标
- 本 Skill 用于：科研项目的代码全生命周期管理，从技术路线设计到代码组织维护
- 能力包含：技术路线设计、算法与代码方案选择、代码结构分析与可视化、项目模板生成
- 触发条件：用户需要规划科研方案、选择算法实现、整理代码结构、创建新科研项目

## 前置准备
- 依赖说明：
  ```
  networkx>=2.8
  graphviz>=0.20
  ```

## 操作步骤

### 意图判断（内部判断，不向用户表达）
- **执行时机**：对话开始时，首先隐式判断用户意图类型
- **判断标准**：基于任务量、系统性和复杂度进行判断
  - **科研项目**：
    - **系统性**：任务具有系统性、综合性，需要多个子任务协同完成
    - **多模块**：需要多个代码模块（数据、模型、训练、评估、可视化等）协同工作
    - **长期性**：需要持续迭代和优化，不是一次性完成
    - **完整流程**：涉及完整的科研实验流程（数据准备→模型设计→训练验证→结果分析→可视化）
    - 示例："我要做一个图像分类研究"、"设计一个推荐系统"、"构建一个时间序列预测系统"
  - **单一任务**：
    - **单一功能**：任务量无需多模块解决，直接一套算法或一个函数可以完成
    - **范围有限**：任务明确、范围较小、可独立完成
    - **无系统性**：不涉及完整的实验流程，只是实现某个具体功能
    - 示例："实现一个数据增强函数"、"写一个LSTM模型"、"修复这个bug"、"写一个排序算法"
- **分流策略**：
  - 科研项目 → 执行"完整科研项目流程"
  - 单一任务 → 执行"单一任务流程"

### 完整科研项目流程

#### 0. 需求澄清（重要前置步骤）
- **执行时机**：在提供任何技术建议之前，必须先确认用户需求的清晰度
- **判断标准**：当用户描述模糊、缺少关键信息时，主动提出澄清问题
- **智能体主导**：通过提问收集以下核心信息（仅限这三方面，不要询问研究方法和指标）：
  - **研究对象**：要研究什么数据？数据类型（图像/文本/音频/视频/表格数据等）？数据规模大概多少？
  - **研究目标**：要解决什么问题？预期达到什么效果？
  - **研究基础**：已有哪些资源？是否有参考论文？是否有现有代码或baseline模型？是否有技术栈偏好（如编程语言、深度学习框架）？
- **输出内容**：
  - 针对缺失信息提出2-3个核心问题（聚焦研究对象、目标、基础）
  - **⚠️ 提出问题后必须停止生成，等待用户回答**
  - **不要继续执行后续步骤1-5，直到用户明确回答了所有问题**
  - 用户回答后，基于明确的信息继续执行步骤1
- **重要原则**：不要询问"任务类型"、"评估指标"、"使用什么算法"等问题，这些应由AI在步骤1中主动分析和建议

#### 1. 科研路线分析（科研视角）
- **智能体主导**：从科研方法论角度分析，不只是技术实现
- **阅读指导**：参考 [references/methodology-guide.md](references/methodology-guide.md) 了解各领域研究方法
- **分析视角**：
  - **科研价值**：每条路线的创新点和学术贡献
  - **实验设计**：验证方法、对照组设置、消融实验
  - **可行性**：实现难度、时间成本、资源需求
  - **风险分析**：潜在失败点和应对策略
- **输出内容**：
  - 问题定义与数据理解
  - 可行科研路线（2-3个备选方案）
  - **路线比较分析（重要）**：
    - 对比每条路线的利弊（科研价值、创新性、实现难度）
    - 分析每条路线的实验设计和验证方法
    - 评估每条路线的学术贡献和发表潜力
  - **⚠️ 提出路线方案后必须停止生成，等待用户选择**

#### 2. 用户选择确认（重要）
- **执行时机**：在步骤1提供多个备选路线后
- **智能体行为**：
  - **⚠️ 必须停止生成，等待用户选择科研路线**
  - 不要继续执行步骤3-8
  - 等待用户明确选择某条路线或提出修改意见
- **用户选择后**：
  - 基于用户确认的路线，继续执行步骤3（文献调研与综述）
  - 如果用户提出新想法，重新分析并再次提供方案

#### 3. 文献调研与综述
- **执行时机**：用户确认研究路线后，在进行算法选择之前
- **智能体主导**：深度搜索相关文献，生成文献综述
- **执行步骤**：
  1. **文献搜索**：使用web_search工具搜索相关文献（学术论文、技术报告、综述文章）
  2. **文献筛选**：按相关性、权威性、时效性（近3-5年）筛选
  3. **文献分析**：提取关键信息（方法、实验、创新点），分类整理，对比分析
  4. **综述生成**：生成Markdown文档，可选调用脚本生成HTML网页
- **输出内容**：
  - **文献综述报告**：研究背景、相关工作分类、方法对比、研究现状、未来方向、参考文献
  - **文献综述网页**（可选）：调用 `scripts/generate_literature_review.py` 生成HTML
- **⚠️ 注意**：文献搜索依赖网络访问，建议用户结合学术数据库补充搜索

#### 4. 算法与代码选择
- **智能体主导**：根据用户确认的科研路线，对比不同算法和实现方案
- **阅读指导**：参考 [references/algorithm-selection.md](references/algorithm-selection.md) 了解算法对比
- **输出内容**：
  - **算法科研适配性对比（重要）**：
    - 算法与科研问题的匹配度
    - 算法对研究目标的支持度（如何帮助实现研究目标）
    - 算法的创新性贡献（能否体现研究创新点）
    - 实现可行性（技术难度、资源需求）
    - 潜在风险（可能的问题和限制）
    - 注意：弱化纯性能指标对比，重点关注科研适配性
  - **选择理由分析（重要）**：
    - 该算法如何支持科研目标
    - 该算法如何体现研究创新点
    - 该算法与科研问题的匹配度分析
    - 说明排除其他算法的原因（基于科研目标而非单纯性能）
  - **代码实现流程列举（重要）**：
    - 列出完整的代码实现流程，包括：
      - 数据加载与预处理
      - 数据增强（如需要）
      - 模型定义
      - 损失函数与优化器
      - 训练循环
      - 验证与评估
      - 结果可视化
      - 模型保存与加载
      - 推理脚本
      - 实验记录与日志
    - 对每个模块说明实现要点
  - **⚠️ 列举代码过程后必须停止生成，询问用户需要生成哪些代码模块**
  - 等待用户选择需要生成的代码模块后，再根据用户选择生成具体代码

#### 5. 建立项目文件夹（重要）
- **执行时机**：在用户选择代码模块之后，开始生成代码之前
- **智能体主导**：根据科研路线和算法选择，创建项目目录结构
- **使用脚本**：调用 `scripts/generate_template.py` 生成标准项目骨架
- **执行步骤**：
  1. 询问用户：是否让AI自动创建项目文件夹？
     - **选项A：AI自动创建** - AI基于科研主题生成项目名称，使用默认路径
     - **选项B：手动指定** - 用户自定义项目名称和路径
  2. 如果选择A（AI自动创建）：
     - AI根据科研主题自动生成项目名称（如：cifar-senet-classification）
     - 使用默认路径（./projects）
     - 向用户确认生成的项目名称和路径
  3. 如果选择B（手动指定）：
     - 询问用户项目名称和存储路径
  4. 确定项目类型（deep-learning/ml/statistics/numerical）
  5. 调用脚本生成项目骨架
  6. 确认项目创建成功
- **输出内容**：
  - 项目目录结构
  - 模块职责划分
  - 配置管理方案
  - 实验跟踪机制
  - 项目路径（便于后续代码生成和查看更新）
  - **代码版本管理策略（重要）**：
    - 初始版本：v1.0.0
    - 文件命名格式：模块名_v版本号.py
    - 示例：dataset_v1.0.0.py, model_v1.0.0.py, trainer_v1.0.0.py
    - 版本递增规则：
      - 小修复（bugfix）：v1.0.0 → v1.0.1
      - 功能更新：v1.0.0 → v1.1.0
      - 重大更新：v1.0.0 → v2.0.0
    - 保留历史版本：所有版本文件都保留，便于回溯和对比
- **⚠️ 项目文件夹建立完成后，向用户确认路径，然后根据用户选择的代码模块生成代码**

#### 6. 代码生成（版本化管理）
- **智能体主导**：根据用户选择的代码模块，生成完整的代码文件
- **生成内容**：完整代码、代码注释、使用示例、测试代码
- **代码标注规范（重要）**：
  - 标注格式：`# TODO: 用户自定义 - <说明>`
  - 标注内容：超参数配置、路径配置、模型参数、数据处理参数
  - 示例：
    ```python
    # TODO: 用户自定义 - 根据实际情况修改数据路径
    DATA_PATH = "./data/cifar-10"
    # TODO: 用户自定义 - 根据硬件调整batch size
    BATCH_SIZE = 128
    ```
- **文件命名**：版本化命名（模块名_v1.0.0.py）
- **输出内容**：
  - 完整代码文件（带用户自定义标注）
  - 代码说明文档
  - **用户自定义提醒**：明确列出必须修改的配置项，特别强调关键配置

#### 7. 代码结构分析与可视化
- **脚本执行**：
  1. 调用 `scripts/analyze_structure.py` 扫描现有代码目录，提取结构信息
  2. 调用 `scripts/visualize_deps.py` 基于结构报告生成依赖关系图
- **智能体分析**：结合可视化结果，评估代码组织质量，提出改进建议
- **输出内容**：
  - 代码结构报告（模块、类、函数列表）
  - 依赖关系图
  - 架构优化建议

#### 8. 代码迭代与维护
- **智能体主导**：在代码迭代过程中保持架构一致性
- **脚本辅助**：定期运行结构分析和依赖可视化，监控代码健康度
- **输出内容**：
  - 修改建议和影响评估
  - 更新后的依赖关系图
  - 代码重构方案（如需要）

### 单一任务流程

#### S0. 单一任务需求澄清（条件性执行）
- **执行时机**：在任务拆解之前，判断用户需求是否足够具体
- **判断标准**：
  - **需求具体**：包含明确的任务描述、输入输出、使用场景
    - 示例："写一个函数，输入是numpy数组，输出是随机裁剪后的图像，裁剪大小为32"
    - 处理：直接进入S1，无需澄清
  - **需求模糊**：缺少关键信息，任务目标不明确
    - 示例："帮我写个数据增强"
    - 处理：追问以下问题
- **追问内容（当需求模糊时）**：
  1. **任务内容目标**：具体要实现什么功能？达到什么效果？
  2. **前提基础**：是否有相关代码、数据、环境要求？
  3. **后续拓展**：是否需要考虑扩展性、后续可能的功能？
- **输出内容**：
  - 针对缺失信息提出2-3个关键问题
  - **⚠️ 提出问题后必须停止生成，等待用户回答**
  - 用户回答后，基于明确的信息继续执行S1

#### S1. 任务理解与拆解
- **智能体主导**：理解用户的具体任务需求，拆解为可执行的子任务
- **分析内容**：
  - 任务核心目标：要实现什么功能
  - 输入输出：明确输入参数和预期输出
  - 依赖关系：需要哪些外部库或现有代码
  - 边界条件：异常情况、边界值处理
- **输出内容**：
  - 任务概述
  - 任务拆解：将复杂任务拆解为2-5个子任务
  - 每个子任务的实现要点
  - 依赖的库和模块

#### S2. 代码实现流程设计
- **智能体主导**：为每个子任务设计详细的代码实现流程
- **设计内容**：
  - 算法选择：选择最适合的算法或方法
  - 数据结构：定义需要的数据结构
  - 函数设计：确定函数接口、参数、返回值
  - 错误处理：设计异常处理机制
- **输出内容**：
  - 代码实现流程图（文字描述）
  - 核心算法说明
  - 函数接口定义
  - 关键代码片段示例
  - 测试用例建议

#### S3. 建立工作文件夹（强制前置步骤）
- **执行时机**：在生成代码之前，必须先建立文件夹
- **智能体主导**：询问用户是否需要创建项目文件夹
- **执行步骤**：
  1. 询问用户：需要创建项目文件夹吗？
     - **选项A：需要创建**（推荐）
     - **选项B：直接在当前目录生成**
  2. 如果选择A（需要创建）：
     - 询问项目名称和存储路径
     - 使用 `mkdir -p` 创建文件夹
     - 确认文件夹创建成功
  3. 如果选择B（直接生成）：
     - 直接在当前目录生成代码文件
- **输出内容**：
  - 文件夹路径（如果创建了文件夹）
  - 代码将保存的位置
- **⚠️ 重要**：代码文件必须保存到文件夹中，不要散落在工作区根目录

#### S4. 代码生成
- **智能体主导**：基于设计的流程生成完整代码
- **生成内容**：
  - 完整的Python代码（或其他语言）
  - 代码注释和文档字符串
  - 使用示例
  - 测试代码
- **代码标注规范（重要）**：
  - 在所有生成的代码中，使用统一注释格式标注需要用户自定义的部分
  - 标注格式：`# TODO: 用户自定义 - <说明>`
  - 常见需要标注的内容：
    - **参数配置**：函数默认参数、阈值、超参数等
    - **路径配置**：输入文件路径、输出路径等
    - **数据格式**：输入数据的字段名、数据类型等
    - **业务逻辑**：特定规则、条件判断等
  - 示例：
    ```python
    def process_data(data_path, threshold=0.5):
        """
        处理数据
        
        Args:
            data_path: TODO: 用户自定义 - 输入数据文件路径
            threshold: TODO: 用户自定义 - 根据业务需求调整阈值
        """
        # TODO: 用户自定义 - 根据实际数据格式读取
        data = load_data(data_path)
        
        # TODO: 用户自定义 - 根据业务规则处理
        result = data > threshold
        
        return result
    ```
- **文件命名**：
  - 使用版本化命名：模块名_v1.0.0.py
  - 示例：data_augmentation_v1.0.0.py, lstm_model_v1.0.0.py
- **输出内容**：
  - 完整代码文件（带用户自定义标注）
  - 代码说明文档
  - 使用指南
  - 测试验证
  - **用户自定义提醒（重要）**：
    - 在对话中明确列出必须修改的配置项
    - 特别强调不修改就无法运行的关键配置（如文件路径、数据导入等）
    - 提醒内容格式：
      ```
      ⚠️ 重要提醒：以下配置项必须修改，否则代码无法运行：

      【必须修改】
      1. 输入文件路径：第X行 - data_path参数
         - 修改为：您的实际输入文件路径
      2. 输出路径：第Y行 - output_path变量
         - 修改为：您的期望输出路径

      【建议修改】
      3. 阈值参数：第Z行 - threshold默认值
         - 当前值：0.5
         - 建议：根据业务需求调整

      📝 查找方法：在代码中搜索"TODO: 用户自定义"可以快速定位所有需要修改的地方
      ```

#### S5. 代码优化与建议
- **智能体主导**：分析生成代码，提供优化建议
- **优化方向**：
  - 性能优化：时间复杂度、空间复杂度
  - 代码质量：可读性、可维护性
  - 最佳实践：遵循Python规范、设计模式
- **输出内容**：
  - 性能分析
  - 优化建议
  - 潜在问题提醒
  - 扩展建议

### 可选分支

#### 完整科研项目场景
- **当需要新建项目**：执行步骤0-5，需求澄清→科研路线分析→用户选择→文献调研与综述→算法与代码选择→建立项目文件夹
- **当需要分析现有项目**：直接执行步骤7，调用分析脚本
- **当需要选择算法**：执行步骤0-4，需求澄清→科研路线分析→文献调研与综述→算法与代码选择
- **当需要整理混乱代码**：执行步骤7-8，基于分析结果重构

#### 单一任务场景
- **当需要实现具体功能**：执行步骤S1-S4，任务理解与拆解→代码实现流程设计→建立工作文件夹→代码生成
- **当需要优化现有代码**：执行步骤S5，代码优化与建议
- **当需要修复bug**：执行步骤S1-S4，问题分析→解决方案设计→建立工作文件夹→代码修复

## 注意事项
- **意图判断优先（内部）**：对话开始时首先隐式判断用户意图类型（完整科研项目 vs 单一任务），选择相应流程，无需向用户表达
- **需求澄清优先（强制）**：在提供任何技术建议之前，必须先通过提问确认用户需求的清晰度
  - 提出澄清问题后**必须停止生成**，等待用户回答
  - **不要继续执行后续步骤**，直到用户明确回答
  - 确保理解准确后再提供技术建议，避免盲目给出不匹配的建议
- **建立文件夹优先（强制）**：在生成任何代码之前，必须先建立工作文件夹
  - 询问用户是否需要创建项目文件夹
  - 确保代码文件保存到专门的文件夹中，不要散落在工作区根目录
  - 科研项目和单一任务都必须遵循此规则
- **代码标注规范（强制）**：在所有生成的代码中必须标注需要用户自定义的部分
  - 使用统一格式：`# TODO: 用户自定义 - <说明>`
  - 标注范围：超参数、路径配置、模型参数、数据处理参数等
  - 目的：让用户快速定位并修改需要调整的配置项
- 充分利用智能体的知识推理能力，技术路线设计和算法选择主要依靠智能体而非脚本
- 脚本仅用于代码结构分析和模板生成，避免过度依赖自动化工具
- 在阅读参考文档时，先阅读目录和概览，按需深入相关章节
- 保持代码架构的迭代式优化，通过定期分析依赖关系监控代码健康度
- 新建项目时优先使用标准模板，保持代码组织的一致性
- 单一任务处理时，注重任务拆解的细致性和代码实现流程的完整性

### 示例1：设计深度学习研究项目的技术路线
```
用户需求：我要做一个图像分类研究

执行流程：
1. 需求澄清：询问数据类型、数据规模、研究基础（baseline/SOTA、技术栈偏好）
2. 科研路线分析：提供2-3条备选路线，从科研价值、实验设计、可行性、风险角度分析
3. 用户选择：选择路线B（ResNet-18 + 注意力机制）
4. 文献调研与综述：搜索相关文献（注意力机制、图像分类），生成文献综述报告
5. 算法与代码选择：算法科研适配性对比、代码实现流程列举
6. 用户选择代码模块：选择模块1、3、4、5、6
7. 建立项目文件夹：AI自动创建或手动指定，使用generate_template.py生成项目骨架
8. 生成代码：根据用户选择的模块生成v1.0.0版本代码（带用户自定义标注）

```
